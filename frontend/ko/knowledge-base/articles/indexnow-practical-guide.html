IndexNow: 실용적인 구현 및 팁

2025년 9월 15일 • 5분 읽기
왜 읽어야 하나요: 검색 엔진에 빠르게 업데이트를 전달하는 방법, 변경된-항목만을 사용하는 시기, IndexNow가 사이트맵 핑과 어떻게 다른지, 일반적인 오류를 피하는 방법.

1) IndexNow란 무엇인가

IndexNow는 URL 변경(생성, 업데이트, 삭제)에 대해 검색 엔진에 즉시 알릴 수 있는 프로토콜입니다. Bing, Yandex, Seznam 및 기타 서비스에서 지원합니다. 장점은 최소한의 인덱싱 지연과 적은 크롤링 노이즈입니다.

2) 키 설정 및 기본 핑

  1. 키(GUID)를 생성하고 YOUR_KEY.txt 파일을 사이트 루트(/.well-known/ 또는 도메인 루트)에 키 문자열과 함께 배치합니다.
  2. URL과 키로 엔드포인트에 핑을 보냅니다. 예시(단일 URL):
POST https://api.indexnow.org/indexnow Content-Type: application/json { "host": "www.serpmonn.ru", "key": "YOUR_KEY", "urlList": [ "https://www.serpmonn.ru/frontend/knowledge-base/articles/utm-complete-guide.html" ] }
일괄 처리의 경우 한 요청에 최대 10,000개의 URL을 지정하거나 변경 로그 파일 URL이 포함된 파일 피드를 사용합니다.

3) 변경된-항목만 모드

마지막 커밋/배포에서 실제로 변경된 파일만 핑하는 아이디어입니다. 이렇게 하면 노이즈와 할당량 사용이 줄어듭니다.

  • VCS(git diff) 또는 빌드 아티팩트에서 변경된 경로 목록을 가져옵니다.
  • 공개 경로로 필터링합니다(node_modules, .git, 개인 폴더 제외).
  • 관련 없는 URL 전송을 피하기 위해 도메인/언어별로 그룹화합니다.
프로덕션에서는 대체 수단을 갖는 것이 유용합니다: 변경된-항목만이 비어 있는 경우(예: 구성에 대한 핫픽스), 중요한 사이트맵에 핑을 보냅니다.

4) IndexNow vs 사이트맵 핑

  • 속도: IndexNow는 업데이트를 더 빠르게 제공하며 크롤링 주기를 기다리지 않습니다. 사이트맵 핑은 사이트맵의 존재를 알리지만 특정 URL의 변경 사항은 알리지 않습니다.
  • 노이즈: IndexNow - 변경된 URL을 대상으로 합니다. 사이트맵 - 전체 맵 스캔(크롤링 예산에 더 비쌉니다).
  • 신뢰성: 사이트맵은 기초로 남아 있습니다. IndexNow는 보완하고 가속화합니다. 둘 다 사용하는 것이 좋습니다.

5) 일반적인 오류

  • 키가 공개 URL을 통해 액세스할 수 없거나 key 매개변수와 일치하지 않습니다.
  • 개인/404 페이지 핑, 표준화되지 않은 매개변수.
  • 동일한 URL의 너무 빈번한 반복(반복을 제한합니다).
  • 다국어 지원 무시(해당 지역화 버전을 핑해야 합니다).

6) FAQ 및 체크리스트

한 번에 몇 개의 URL을 보내야 합니까?

일괄 처리당 최대 10,000개 - 유형/언어별로 최적으로 그룹화합니다.

삭제를 핑해야 합니까?

예 - 삭제된 페이지의 URL을 전달하여 인덱스 해제를 가속화합니다.

작동하는지 확인하는 방법은?

API 응답 로그(200/202), 서버 로그의 크롤링 급증, 새로운 URL의 더 빠른 출현.

  • 키가 배치되고 HTTPS를 통해 액세스할 수 있습니다.
  • CI/CD에서 변경된-항목만 경로를 수집합니다.
  • 중복 제거 및 반복 제한.
  • 빈 일괄 처리 시 사이트맵 핑으로 대체.
: IndexNow는 새롭고 변경된 페이지의 인덱싱을 크게 가속화합니다. 기존 사이트맵과 결합하면 SEO 최적화를 위한 강력한 도구입니다.
